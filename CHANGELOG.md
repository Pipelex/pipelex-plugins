## [Unreleased]

### Added

- Repo bootstrap: multi-target build tooling (`pyproject.toml`, `Makefile`), Apache-2.0 license, and project documentation, ported and adapted from the `mthds-plugins` scaffolding without any local-CLI lifecycle machinery.
- Multi-target build system: the `scripts/gen_skill_docs.py` Jinja2 renderer and `scripts/check.py` consistency checks, `targets/` (prod, codex, mistral-vibe, and a trimmed `defaults.toml`), the shared MTHDS language reference templates, and unit tests. `frontmatter.md.j2` is an include-only partial. None of the predecessor's install/upgrade/env-check switches are carried over.
- Marketplace and manifests: Claude `.claude-plugin/` (plugin base + marketplace), Codex `.codex-plugin/` plugin base and `packaging/codex-marketplace.json` with its generated `.agents/plugins/` discovery copy, and per-target `plugin.json` generation. Marketplace/version consistency is enforced by `make check`.
- First skill: `pipelex-explain` (ported and stripped from `mthds-explain`) — a pure read-and-explain skill that reads an `.mthds` bundle, identifies its components, traces the execution flow, and presents a plain-language explanation with a text flow diagram. No CLI dependency: the env-check step, the validate/graph steps, and the CLI-required framing are gone. Grounded on the shared MTHDS language references. Rendered across the Claude, Codex, and Mistral Vibe targets.
- **CLI-free wasm+API validation hooks on all targets** (Claude `PostToolUse`, Codex `PostToolUse` over `apply_patch`, Mistral Vibe `after_tool`). Each target ships a thin fail-open wrapper script that runs the shared vendored `hooks/check.mjs` bundle (built in `pipelex-sdk-js`, provenance header included) with a per-platform flag: lint and format run **locally** via the `@pipelex/tools-wasm` engine inlined in the bundle (offline, no credentials; format writes back in place), and the full bundle verdict comes from `POST /v1/validate` through `@pipelex/sdk` (`allow_signatures: true`, `.mthds` files gathered recursively under the edited file's parent dir with caps, server-rendered Markdown forwarded verbatim as the block reason, non-blocking `pending_signatures` nudge). Per-platform input/output: Claude reads `tool_input.file_path` and speaks block/`hookSpecificOutput`; Codex parses the `apply_patch` envelope (several `.mthds` files per patch — outcomes merged, any block wins) with the same output dialect, wired as `${PLUGIN_ROOT}/hooks/validate-mthds-codex.sh` from `codex-hooks.json`; Vibe reads the AfterToolInvocation payload and speaks deny/`hook_specific_output.additional_context`. Fail-open everywhere: no Node → the whole hook passes silently; no `PIPELEX_API_KEY` / API unreachable / timeout / any non-2xx (including a network-sandboxed Codex session) → the local lint/format verdicts still apply and only the validate stage is skipped. Nothing shells out to `plxt` or `mthds-agent` anymore. Hook timeouts set to 15 s (the one network call has its own 10 s ceiling inside the bundle). The bundle ships as a static hook asset (`templates/hooks/assets/check.mjs`, copied verbatim by the build via `STATIC_HOOK_ASSETS_BY_PLATFORM`); `make vendor-hook` re-vendors it from the sibling `pipelex-sdk-js` checkout (the wasm engine rides in as the published `@pipelex/tools-wasm` npm dependency) and `make check` fails on a missing or stale copy; `make check` also enforces the Mistral Vibe hook artifacts.
- Build documentation: `docs/build-targets.md` describing the multi-target build, and `docs/hooks.md` describing the validation pipeline, the CLI-free silent-pass posture, per-platform wiring, and the Codex hook trust note (the `hooks` feature is Stable and on by default in Codex 0.141+, so there is nothing to enable — the bundled hook loads on its own and only needs first-run trust).
- New skill: `pipelex-design` (ported from `mthds-recursive`, renamed to design-oriented vocabulary throughout — same top-down stepwise-refinement methodology). It captures a whole job as one `PipeSignature`, refines one signature at a time into operators and controllers, and keeps the library valid at every layer. Fully CLI-free: validation goes through the `mthds_validate` MCP tool (whole-bundle submission; verdict discipline branched on the structured fields, backlog read from the Markdown summary's pending-signatures section), the finalize gate is the runnable verdict (`is_runnable: true` + empty `pending_signatures` — the MCP always permits signatures, so there is no separate strict pass), the Deliver phase shows the method's input template via the `mthds_inputs` MCP tool and hands off to `/pipelex-inputs`, and lint/format is left to the plugin's PostToolUse hook. When the MCP tools are absent or the server is misconfigured, the skill stops with a one-line setup instruction instead of skipping validation. Ships with the `design-cheat-sheet.md` reference (ported from `recursive-cheat-sheet.md`; the lenient/strict CLI validation section is replaced by the MCP runnable-gate semantics). Rendered across all three targets.
- New skill: `pipelex-inputs` (ported from `mthds-inputs`) — the single entry point for preparing a method's `inputs.json`: placeholder template, synthetic test data, user-provided files, or a mix, with the same mode-selection and file-matching heuristics as its predecessor. Fully CLI-free: the whole schema-extraction step is one `mthds_inputs` MCP tool call (whole-bundle submission, structured-verdict branching), and the skill stops with a setup instruction when the tool is absent or the server is misconfigured. `inputs.json` now uses the **light** template shape everywhere — bare values for scalar inputs (string/number/boolean/date/URL-or-path), content dicts for structured concepts, lists for multiples; the ceremonial `{concept, content}` envelope is gone from all examples. Dropped relative to the predecessor: the remote-storage upload branches, the run-a-pipeline image-synthesis section, the offer-to-run tail, and the Pipelex-install-bound Python invocations (document generation now uses environment-neutral `uv run --with <package> python`). The native content structures section is replaced by a light-shape value table linking the shared reference. `pipelex-design`'s Deliver phase hands off to it. Rendered across all three targets.
- Plugin-declared MCP server: the generated Claude and Codex manifests now carry an inline `mcpServers` entry for the `pipelex-mcp` server (streamable HTTP; tools `mthds_validate` and `mthds_inputs`), sourced from the new `mcp_server_url` template variable. Claude wraps the URL as `${PIPELEX_MCP_URL:-<url>}` so a session-start env var switches between the baked default and a dev server without a rebuild; Codex gets the literal URL (it does no env expansion in MCP config — the dev override is a same-named `[mcp_servers.pipelex]` entry in `~/.codex/config.toml`, which outranks the plugin declaration; verified against Codex 0.144.4, where the tools reach the model as `mcp__pipelex__mthds_validate` / `mcp__pipelex__mthds_inputs`). The baked default is a placeholder until `pipelex-mcp` is deployed. Vibe manifests get no entry (plugin-bundled MCP support unverified there — manual registration is the documented interim). See `docs/decisions.md`.
- Codex 0.144.4 verification and dev-loop fixes: the `.mthds` validation hook was verified in live Codex sessions (broken `.mthds` blocked with lint diagnostics relayed to the model, valid-but-unformatted write formatted in place, non-`.mthds` patches silent), and the `make codex-refresh` / `codex-use-*` targets were adapted to the 0.144 plugin cache model — installed plugins run from a cache copy, `codex plugin marketplace upgrade` is Git-only now, so local edits propagate via an idempotent `codex plugin add`.
