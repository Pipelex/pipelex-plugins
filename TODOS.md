# TODOS — Mistral Vibe stable hooks migration

Tracker for adopting the **stable** Mistral Vibe hooks API in `pipelex-plugins`. Use the checkboxes to track progress; update the checkpoint notes at each phase boundary so a fresh session can pick up cleanly.

## Context (read this on cold start)

**What changed upstream.** Mistral Vibe hooks are now **stable** (previously experimental). Docs: <https://docs.mistral.ai/vibe/code/cli/hooks>. Key facts verified against those docs:

- Three hook types: `post_agent` (after an agent turn), `pre_tool` (before a tool runs), `post_tool` (after a tool runs).
- **Breaking renames** from the experimental version: `post_agent_turn` → `post_agent`, `before_tool` → `pre_tool`, `after_tool` → `post_tool`.
- **No opt-in flag anymore** — `enable_experimental_hooks = true` in `~/.vibe/config.toml` is obsolete.
- Declared in `./.vibe/hooks.toml` (project-level, trusted folders only) or `~/.vibe/hooks.toml` (user-level). Project entries override user entries when `name` matches; subagents inherit. There is still **no plugin-bundled hook mechanism** — manual wiring into `hooks.toml` remains the install story.
- TOML fields unchanged from what we ship: `name`, `type`, `match` (fnmatch or `re:` regex, case-insensitive, required for tool hooks), `command`, `timeout` (default 60 s), `strict`, `description`.
- `post_tool` stdin payload: common fields (`session_id`, `parent_session_id`, `transcript_path`, `cwd`, `hook_event_name`) plus `tool_name`, `tool_call_id`, `tool_input`, `tool_status`, `tool_output`, `tool_output_text`, `tool_error`, `duration_ms`.
- Response dialect unchanged: `{"decision":"deny","reason":…}` to block; `{"decision":"allow","hook_specific_output":{"additional_context":…}}` to append context; exit 0 + empty stdout = passthrough; under `strict = true` failures become denials.

**Current state of our repos:**

- `pipelex-plugins` (this repo) ships a `mistral-vibe` target (`pipelex-vibe/`) whose hook config uses the **old experimental** `type = "after_tool"`. The validation logic lives in the shared vendored `hooks/check.mjs` bundle (source: `../pipelex-sdk-js/src/hooks/check-core.ts` + `claude-mthds-check.ts`, re-vendored via `make vendor-hook`); the Vibe wrapper is `templates/hooks/check-mthds-vibe.sh.j2`. `check.mjs --platform=vibe` gates on `tool_status === "success"`, extracts the path from `tool_output.file`/`.path` or `tool_input.file_path`/`.path` resolved against `cwd`, and already speaks the deny / `hook_specific_output.additional_context` dialect — so the **response side needs no code change**; the payload side needs live verification (see Phase 2).
- `../mthds-plugins` has **no Mistral Vibe target at all** — its targets are `prod`, `codex`, `dev`, `sandbox`. (The old `mthds-vibe` name there was a *skill*, since renamed `mthds-recursive`; it has nothing to do with Mistral Vibe.) **Out of scope for now** — see "Deferred" at the bottom.

**Where every `after_tool` / experimental reference lives in pipelex-plugins** (grep `after_tool\|AfterToolInvocation\|enable_experimental_hooks` to re-verify):

- `templates/hooks/vibe-hooks.toml.j2` — the hook `type` itself
- `templates/hooks/check-mthds-vibe.sh.j2` — header comments (AfterToolInvocation wording)
- `scripts/check.py` — `check_vibe_target_artifacts` asserts `type = "after_tool"`
- `tests/unit/test_check.py`, `tests/unit/test_gen_skill_docs.py` — fixtures with `after_tool`
- `README.md` — Vibe install section: `enable_experimental_hooks` flag, `after_tool` snippet, "Requires Mistral Vibe 2.15.0+" line
- `CLAUDE.md`, `docs/hooks.md`, `docs/build-targets.md`, `CHANGELOG.md` — prose mentions
- Generated outputs `pipelex-vibe/hooks/*` — regenerated by `make build`, never hand-edited

## Phase 2 — verify `check.mjs` payload compatibility against live stable Vibe

The bundle's extractor was written against the experimental payload. The stable docs list the same fields we rely on (`tool_status`, `tool_input`, `tool_output`, `cwd`), but the *values* need live confirmation. Source lives in `../pipelex-sdk-js/src/hooks/` (`check-core.ts`: `extractVibeMthdsFilePath`, `formatOutcome`); if it changes, rebuild there and re-vendor here with `make vendor-hook`, then `make build`.

- [x] Vibe 2.21.0 (the stable-hooks release) is installed locally. Payload captured from the **installed package source itself**: instead of a live agent run (blocked by session permissions — see note below), payloads were generated with Vibe's own `PostToolInvocation.model_dump_json()` (`vibe/core/hooks/models.py`), byte-identical to what `hooks/executor.py` pipes to hook stdin.
- [x] Verified against source + generated payloads: `tool_status` for a successful call is `"success"` (`Literal["success","failure","cancelled"]`; set in `agent_loop/_loop.py` `_invoke_tool`); `hook_event_name` serializes as `"post_tool"`; tool names are snake_cased class names → exactly `edit` / `write_file` (match regex correct); `edit`'s `tool_output.file` is the resolved absolute path; `write_file`'s output field is `file_path` (not in the `tool_output` fallback chain) but `tool_input.file_path` covers it for both tools; `cwd` is Vibe's process cwd and relative paths resolve against it.
- [x] Responses verified end-to-end through the shipped `check-mthds-vibe.sh` with the generated payloads: invalid `.mthds` → `{"decision":"deny","reason":…}` with lint diagnostics; format write-back rewrites the file in place; `tool_status: "failure"` payload passes silently; relative `tool_input.file_path` resolves against payload `cwd` even when the hook runs from another directory. The deny/allow dialect matches Vibe's `HookStructuredResponse` schema and `_post_tool.py` handler exactly (deny replaces `tool_output_text` with our reason; allow+`additional_context` appends). The pending-signatures `additional_context` path wasn't exercised live (needs `PIPELEX_API_KEY`); its output shape is schema-verified.
- [x] No divergence found → no `extractVibeMthdsFilePath` / dialect fix, no re-vendor. Findings recorded in `docs/hooks.md` ("The Vibe hook — payload verification"), including the `create_subprocess_shell` cwd nuance (relative hook `command` resolves against the project dir, matching the README's absolute-path instruction).
- [x] "AfterToolInvocation" wording updated to stable `post_tool` in `../pipelex-sdk-js/src/hooks/check-core.ts` + `claude-mthds-check.ts` (comment-only — bundle unchanged, no re-vendor; rides along with the next rebuild).
